# VoxLibris Studio — roadmap стабилизации Icecast baseline

Дата: 2026-04-28

## Цель

Вернуться к простой и предсказуемой рабочей точке VoxLibris Studio на базе Icecast, не продолжая сейчас ни WebRTC-ветку, ни segment-модель как попытку решить pause/privacy.

## Принципы выполнения

- идти маленькими безопасными шагами;
- не ломать уже работающее поведение;
- сохранять только проверенные изменения;
- не смешивать stabilization с новой архитектурой;
- после каждого шага иметь понятную smoke-проверку.

## Roadmap

### 1. Откатить незавершённый segment-эксперимент

Что сделать:

- вернуть `server/routes/studio-stream.ts` к простой baseline-модели записи без segment concat-контура;
- убрать или отложить segment-specific код в `server/lib/studio-recording-storage.ts`, если он больше не нужен для baseline;
- не трогать полезные уже подтверждённые фиксы, не связанные с segment-моделью.

Результат:

- ingest и запись снова имеют минимально необходимую сложность;
- код соответствует реально выбранной стратегии.

### 2. Сохранить подтверждённые стабильные исправления

Сохраняем как часть baseline:

- фикс завершения сессии, при котором `endSession()` выключает и `isLive`, и `isActive`;
- Redis intent-store для разведения `pause` и `end`;
- совместимость админского parser'а записей с текущим именованием файлов;
- текущее рабочее поведение `start -> pause -> resume -> end`.

Результат:

- не теряются уже добытые рабочие улучшения;
- rollback не превращается в откат к более сломанному состоянию.

### 3. Зафиксировать ограничения Icecast как осознанный техдолг

Нужно явно задокументировать:

- live-лаг у listeners порядка 10-15 секунд;
- при `pause` слушатель ещё может некоторое время слышать поток из-за buffering;
- голос, произнесённый в момент `pause`, может долетать в live и в запись;
- это ограничение текущего transport path, а не только локальный баг финализации файла.

Результат:

- команда перестаёт ожидать от текущего контура privacy-гарантий, которых он архитектурно не даёт.

### 4. Повторно прогнать smoke-test стабильного сценария

Проверить вручную:

- `start -> pause -> resume -> end`;
- listener корректно подключается к live;
- после `end` listener больше не остаётся в состоянии "эфир на паузе";
- сессия закрывается предсказуемо и без новых регрессий.

Результат:

- есть подтверждение, что после cleanup baseline действительно рабочий.

### 5. Отдельно проверить контур записей и админку

Проверить:

- файл записи создаётся;
- запись видна в `/admin/recordings`;
- запись скачивается и проигрывается;
- формат имени файла остаётся совместимым с текущей админской логикой.

Результат:

- baseline подтверждён не только по live, но и по post-session артефактам.

### 6. Вынести latency tuning в отдельную ограниченную задачу

Что допускается:

- точечная проверка настроек Icecast, ffmpeg и buffering-пути;
- попытка уменьшить лаг без смены архитектуры;
- только best-effort оптимизация, без обещания privacy-fix.

Что не делать в этой задаче:

- не переходить на WebRTC;
- не возвращаться к segment-модели как к основной стратегии;
- не раздувать объём в новый streaming rewrite.

Результат:

- latency можно попробовать уменьшить, не ломая baseline и не расползаясь по scope.

### 7. Заморозить checkpoint после подтверждения smoke

После прохождения ручных проверок:

- зафиксировать кодовую точку как рабочий Icecast baseline;
- от неё уже отдельно вести будущие экспериментальные ветки;
- не смешивать stabilization-ветку с future architecture track.

## Критерии успеха

Считать roadmap выполненным, когда одновременно выполняются все условия:

- в коде больше нет незавершённого segment-контура как активной стратегии baseline;
- `start -> pause -> resume -> end` работает без регрессии;
- после `end` listener не остаётся в ложном paused-state;
- запись появляется в админке и доступна для скачивания/прослушивания;
- ограничения Icecast зафиксированы как осознанный техдолг;
- WebRTC оставлен как отложенный future track, а не текущая задача.
