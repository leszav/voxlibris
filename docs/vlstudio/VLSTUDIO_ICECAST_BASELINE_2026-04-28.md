# VoxLibris Studio — baseline текущего стабильного Icecast-контура

Дата фиксации: 2026-04-28

## Назначение документа

Зафиксировать, какое состояние VoxLibris Studio сейчас считается рабочим baseline, если цель — надёжно поддерживать live-чтение на Icecast без продолжения архитектурных поворотов в WebRTC или segment-recording.

## 1. Что считается baseline сейчас

Текущий baseline — это pragmatic live-контур, в котором:

- чтец ведёт эфир через существующий Icecast pipeline;
- слушатели подключаются к live-потоку как к streaming delivery слою;
- сервер сохраняет запись эфира;
- админка читает записи с диска и показывает их в `/admin/recordings`;
- pause/resume/end работают в рамках ограничений Icecast buffering.

## 2. Что подтверждено как рабочее

По уже проверенному сценарию:

- `start -> pause -> resume -> end` работает;
- после полного `end` listener больше не видит ложное состояние "эфир на паузе";
- запись после эфира появляется в админке;
- parser админки совместим как минимум с текущим форматом `${sessionId}.mp3` и со старым форматом с timestamp.

## 3. Какие исправления входят в baseline

### 3.1. Исправление завершения сессии

В серверной логике завершения сессии baseline включает исправление, при котором `endSession()` переводит сессию в неактивное и не-live состояние одновременно.

Практический эффект:

- после завершения эфира listener не застревает в состоянии paused-live;
- полное завершение отделено от временной паузы.

### 3.2. Redis intent-store для `pause` и `end`

Baseline включает intent-модель по `sessionId`, где сервер различает:

- `pause`
- `end`

Практический эффект:

- controlled shutdown ingest-контура не трактуется одинаково для паузы и для полного завершения;
- финализация записи лучше согласована с фактическим намерением жизненного цикла сессии.

### 3.3. Исправление видимости записей в админке

Baseline включает совместимость parser'а имён файлов записей с текущим naming.

Практический эффект:

- записи не исчезают из `/admin/recordings` только из-за несовпадения ожидаемого формата имени файла.

## 4. Что не входит в baseline

Следующее не считается частью стабильной выбранной стратегии на текущем этапе:

- segment-recording как решение pause/privacy;
- concat segments на завершении как обязательный production path;
- возврат к WebRTC/mediasoup в рамках текущего stabilization-прохода;
- попытка добиться строгих privacy-гарантий поверх текущего Icecast buffering-контура.

## 5. Осознанные ограничения baseline

На этом baseline принимаются следующие ограничения:

- у live-доставки есть заметный лаг порядка 10-15 секунд;
- после нажатия `pause` listeners могут ещё короткое время слышать уже буферизованный поток;
- голосовой фрагмент, произнесённый вблизи момента паузы, может попасть в live и в итоговую запись;
- это не локальная мелкая ошибка UI, а следствие текущего transport/buffering path.

## 6. Что можно делать поверх этого baseline

Допустимо:

- точечно чистить код до более простой baseline-формы;
- повторять smoke-проверки;
- улучшать устойчивость dev/start/restart сценариев и чистый startup контейнеров;
- отдельно и ограниченно пробовать latency tuning без смены архитектуры;
- улучшать документацию и наблюдаемость.

Нежелательно без отдельного решения:

- продолжать segment-ветку по инерции;
- смешивать stabilization с будущей WebRTC-подготовкой;
- делать широкий streaming rewrite рядом с уже рабочим контуром.

## 7. Контрольная формулировка

Текущий baseline VoxLibris Studio — это рабочий, но ограниченный Icecast-контур, пригодный для дальнейшей стабилизации и эксплуатации как осознанный промежуточный этап.

Его задача сейчас — быть предсказуемым и воспроизводимым, а не решать все long-term проблемы live-аудио.
