# VoxLibris Studio — почему сейчас выбран Icecast и как готовить трек WebRTC return

Дата: 2026-04-28

## 1. Короткий вывод

Сейчас VoxLibris Studio остаётся на Icecast не потому, что WebRTC как идея отвергнут, а потому что для текущего deploy-окружения и текущей стадии продукта Icecast даёт более реалистичную operational-стоимость.

Выбор сделан в пользу:

- предсказуемости;
- меньшего объёма изменений;
- сохранения уже работающего пользовательского сценария;
- снижения инфраструктурного риска на текущем этапе.

## 2. Почему не продолжаем сейчас WebRTC

### 2.1. Причина не в одной только разработке, а в эксплуатации

По уже проведённой проверке стало ясно:

- старый backend foundation под WebRTC/mediasoup в проекте когда-то уже существовал;
- главный барьер возврата — не сама возможность написать код, а сложность production/deploy-контура;
- на текущем сервере и в текущей схеме запуска нет заранее подготовленного простого operational path под mediasoup transports и сопутствующую сетевую конфигурацию.

### 2.2. Текущий продуктовый приоритет другой

Прямо сейчас важнее:

- иметь стабильный baseline для live-чтения;
- не ломать уже работающий user flow;
- не расползтись в многонедельный инфраструктурный трек;
- не смешивать стабилизацию с новой транспортной архитектурой.

### 2.3. WebRTC сейчас был бы дорогим контекстным переключением

Возврат к WebRTC в данный момент означал бы сразу несколько параллельных задач:

- пересборка transport-контура;
- уточнение client flow;
- пересмотр deploy-схемы;
- сетевые и runtime-проверки;
- новая серия reliability и recovery тестов.

Это не маленький безопасный шаг.

## 3. Почему Icecast выбран именно сейчас

### 3.1. Icecast уже встроен в текущий живой контур

Уже существует рабочая цепочка:

- браузерный захват микрофона;
- server-side ingest route;
- `ffmpeg` publish в Icecast;
- listener playback;
- серверная запись эфира;
- административный доступ к записям.

То есть Icecast — это не теория и не заготовка, а реально работающий контур.

### 3.2. Icecast позволяет удержать scope под контролем

На текущем этапе Icecast полезен тем, что позволяет:

- чинить конкретные дефекты в уже существующем потоке;
- сохранять уже рабочий UX;
- стабилизировать dev/start/restart и startup-контур;
- не открывать сразу новый большой фронт по инфраструктуре.

### 3.3. Ограничения Icecast осознаны и приняты

Решение не означает, что ограничений нет. Наоборот, они признаны прямо:

- заметная latency;
- buffering после pause;
- отсутствие строгой privacy-гарантии в паузе;
- часть проблем лежит в transport nature самого решения, а не в одном баге приложения.

Именно поэтому Icecast выбран как baseline, а не как финальная идеальная архитектура.

## 4. Что считать официальным техдолгом Icecast-контура

Нужно считать формализованным техдолгом:

- задержку live-доставки для listeners;
- негарантированное мгновенное "затухание" после pause;
- риск попадания ближайшего к pause голосового маркера в live и запись;
- зависимость качества UX от buffering и особенностей текущего стримингового пути.

Этот техдолг не должен маскироваться под "почти решённую мелочь".

## 5. Что такое `WebRTC return` в этом проекте

`WebRTC return` — это не немедленный rewrite, а отдельный будущий трек, который стартует только после стабилизации текущего baseline и после явного решения, что команда готова оплатить инфраструктурную сложность.

В проекте уже существуют ветки с полноценной WebRTC/mediasoup реализацией VoxLibris Studio.

Реализованные файлы:

- медиапотоки, transport layer и room-модель — хороший reference для архитектуры;
- client-side WebRTC signaling — потребует пересмотра под текущий client codebase;
- deploy-конфигурация для mediasoup — потребует отдельной инфраструктурной работы.

## 7. Условия, при которых имеет смысл возвращаться к WebRTC

Возврат становится рациональным, если одновременно выполняются условия:

- Icecast baseline стабилен и не требует постоянного аварийного вмешательства;
- есть отдельное окно на infra/deploy работу, а не только на app-code;
- есть готовность отдельно проверить UDP/networking и runtime-ограничения;
- есть ясное product-основание, зачем платить этой сложностью: меньшая latency, лучшая pause/privacy-модель, более живой listener experience.

## 8. Подготовленный отложенный трек `WebRTC return`

Когда команда вернётся к этой теме, трек должен выглядеть так:

### Phase 1. Использовать найденный исторический фундамент как reference

- изучить файлы в ветке `dev_08.02.26` / `dev_12.02.26`;
- взять архитектурные паттерны и transport-модель;
- не пытаться сразу resurrect-ить код без новой оценки.

### Phase 2. Проверить инфраструктурную готовность

- выделить требования по сети и transport'ам;
- проверить, что deploy-окружение допускает нужную схему;
- отдельно оценить operational blast radius и мониторинг.

### Phase 3. Спроектировать минимальный pilot

- только audio-only;
- один чтец -> много listeners;
- без лишних feature-ответвлений;
- с явной fallback-стратегией.

### Phase 4. Подготовить migration decision

- сравнить pilot с Icecast baseline по latency, stability и operational cost;
- принять решение на фактах, а не на ожиданиях.

## 9. Что не делать под видом `WebRTC return`

Не стоит:

- тихо продолжать mediasoup-работы внутри stabilization-ветки;
- смешивать latency-fixes на Icecast с фактической реализацией нового transport stack;
- обещать privacy-fix без инфраструктурной готовности;
- принимать частично восстановленный старый код за готовое решение.

## 10. Итоговая формулировка решения

Текущее решение проекта:

- **сейчас**: остаёмся на Icecast, стабилизируем baseline, документируем ограничения как техдолг;
- **исторический reference**: WebRTC-реализация есть в ветках `dev_08.02.26` / `dev_12.02.26`, используется как архитектурный ориентир;
- **позже**: возвращаемся к WebRTC только как к отдельному осознанному architecture track с инфраструктурной подготовкой.

Это не отказ от будущего развития. Это выбор правильного порядка работ.
