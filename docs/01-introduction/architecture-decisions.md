# Архитектурные решения

## Введение

Этот документ описывает ключевые архитектурные решения, принятые при разработке платформы VoxLibris. Он служит как справочное руководство для команды разработчиков и как историческая запись обоснования архитектурных решений.

## 1. Выбор стека технологий

### Контекст
Необходимо выбрать стек технологий, который будет поддерживать масштабируемость, безопасность и возможность быстрой разработки новых функций.

### Решение
- **Frontend**: React 19 + TypeScript + Vite + Tailwind CSS
- **Backend**: Express.js + TypeScript + Node.js 20+
- **Database**: PostgreSQL 14+ + Drizzle ORM
- **File Storage**: AWS S3-совместимое хранилище (MinIO)
- **Real-time communication**: WebSocket (Socket.IO)

### Обоснование
React был выбран как самый популярный фронтенд-фреймворк с большим сообществом и экосистемой. TypeScript обеспечивает безопасность типов на этапе компиляции. Express.js был выбран как легковесный и гибкий backend-фреймворк. PostgreSQL выбран как мощная и надежная реляционная база данных. Drizzle ORM выбран за его типизированность и совместимость с TypeScript. S3-совместимое хранилище обеспечивает масштабируемое и надежное хранение файлов. WebSocket необходим для реального времени в сессиях чтения.

## 2. Архитектурный стиль

### Контекст
Необходимо выбрать архитектурный стиль, который обеспечит модульность, тестируемость и масштабируемость приложения.

### Решение
Применение многоуровневой архитектуры с использованием шаблона Repository:
- Presentation Layer (client/src)
- API Layer (server/routes)
- Business Logic Layer (server/services)
- Data Access Layer (server/repositories)
- Infrastructure Layer (server/lib, server/config)

### Обоснование
Такой подход обеспечивает четкое разделение ответственностей, облегчает тестирование и поддержку кода, а также позволяет легко заменять компоненты на разных уровнях.

## 3. Аутентификация и авторизация

### Контекст
Необходимо обеспечить безопасную аутентификацию пользователей и контроль доступа к ресурсам.

### Решение
Использование JWT-токенов с refresh-токенами, хранящимися в httpOnly cookie:
- Access Token: короткоживущий (15-30 минут)
- Refresh Token: долгоживущий (7-30 дней), хранится в БД
- HttpOnly cookie для защиты от XSS-атак
- Проверка токенов через middleware

### Обоснование
JWT-токены обеспечивают масштабируемость без необходимости хранения сессий на сервере. Refresh-токены обеспечивают безопасное обновление access-токенов. HttpOnly cookie защищает от XSS-атак. Хранение refresh-токенов в БД позволяет отзывать их при необходимости.

## 4. Система клубов и сообществ

### Контекст
Платформа должна поддерживать создание и управление книжными клубами с различными уровнями доступа.

### Решение
Иерархия ролей в клубах:
- Владелец: полный контроль
- Модератор: управление участниками и контентом
- Участник: участие в сессиях
- Гость: ограниченный доступ

### Обоснование
Такая структура обеспечивает гибкость в управлении клубами и позволяет масштабировать платформу до крупных сообществ с различными уровнями участия.

## 5. Система чтения и сессий

### Контекст
Необходимо обеспечить синхронизацию чтения между участниками сессии и поддерживать как индивидуальное, так и коллективное чтение.

### Решение
- WebSocket для синхронизации прогресса в реальном времени
- Разделение на индивидуальные и клубные сессии
- Поддержка синхронизации с аудио для профессиональных записей

### Обоснование
WebSocket обеспечивает минимальные задержки для синхронизации, что важно для совместного чтения. Разделение сессий позволяет оптимизировать ресурсы и управлять доступом.

## 6. Безопасность файлов

### Контекст
Платформа позволяет пользователям загружать файлы книг, что требует проверки безопасности.

### Решение
- Проверка расширения файла (.epub, .fb2)
- Проверка MIME-типа
- Проверка magic numbers для подлинности формата
- Ограничение размера файла (50MB)
- Хранение в S3-совместимом хранилище

### Обоснование
Многоуровневая проверка обеспечивает защиту от загрузки вредоносных файлов. Ограничение размера предотвращает DoS-атаки. Хранение в облаке обеспечивает масштабируемость и надежность.

## 7. Система уведомлений

### Контекст
Пользователям необходимо уведомлять о событиях, но при этом избегать избыточности.

### Решение
- Поддержка нескольких каналов: email, push, внутренние уведомления
- Настройка пользователем типов уведомлений
- Батчевая отправка для производительности
- Система отказов от рассылки

### Обоснование
Разные пользователи предпочитают разные каналы уведомлений. Настройка позволяет персонализировать опыт. Батчевая отправка оптимизирует производительность.

## 8. Модерация контента

### Контекст
Необходимо обеспечить качество контента и соблюдение правил платформы.

### Решение
- Комбинация автоматической фильтрации и ручной модерации
- Система жалоб пользователей
- Рейтинговая система для пользователей и контента
- Логирование модерационных действий

### Обоснование
Автоматическая фильтрация обрабатывает основной объем, а ручная модерация используется для сложных случаев. Рейтинговая система стимулирует качественное поведение.

## 9. Система аудио и записей

### Контекст
Платформа должна поддерживать как живое чтение, так и предварительно записанные версии.

### Решение
- Поддержка различных форматов (MP3, M4A, FLAC, OGG)
- Синхронизация с текстом
- Стриминг для больших файлов
- Качество настройки для разных условий подключения

### Обоснование
Поддержка нескольких форматов обеспечивает совместимость с разными устройствами. Синхронизация с текстом улучшает опыт пользователей. Стриминг позволяет работать с большими файлами.

## 10. Планирование и расписание

### Контекст
Пользователи и организаторы клубов должны иметь возможность планировать сессии чтения.

### Решение
- Интеграция с календарем
- Поддержка повторяющихся событий
- Уведомления и напоминания
- Учет часовых поясов

### Обоснование
Планирование сессий улучшает вовлеченность пользователей. Учет часовых поясов важно для глобальной платформы. Интеграция с календарем упрощает использование.

## 11. Система аналитики

### Контекст
Необходимо собирать данные об использовании для улучшения платформы и принятия решений.

### Решение
- Сбор анонимизированных метрик
- Расчет KPI (коэффициент удержания, DAU/MAU и т.д.)
- Персонализированные рекомендации
- Соответствие требованиям GDPR

### Обоснование
Аналитика позволяет принимать обоснованные решения. Анонимизация обеспечивает конфиденциальность. Персонализация улучшает опыт пользователей.
