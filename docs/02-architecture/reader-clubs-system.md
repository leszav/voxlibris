# Система клубов чтецов

**Статус:** Current  
**Дата обновления:** 2026-06-14

Клубы чтецов — отдельный продуктовый сценарий VoxLibris для live-чтения книги владельцем клуба и прослушивания эфира участниками. Реализация переиспользует базовую инфраструктуру клубов, reading sessions, VoxLibris Studio, приглашений, записей и аналитики, но права доступа и UI-сценарии выделены отдельно от обычных клубов.

## Доменная модель

- Клуб чтецов определяется через `clubs.type = 'reader-led'`.
- `clubs.owner_id` — единственный чтец-владелец клуба.
- Слушатели остаются обычными участниками клуба и не получают роль второго владельца, чтеца или модератора без отдельного сценария.
- Клуб чтецов всегда приватный: при создании `isPrivate = true`, переключатель приватности в UI блокируется.
- Создание клуба чтецов доступно только пользователю с разрешением `canCreateReaderLedClubs` или администратору/модератору согласно серверным проверкам.

## Основные файлы реализации

Сервер:

- `server/club-routes.ts` — базовые маршруты клубов, создание клубов, приглашения, live-readers, owner-only analytics.
- `server/club-reader-routes.ts` — сценарии чтения/доступа к клубному ридеру.
- `server/lib/reader-club-access.ts` — серверные проверки доступа для reader-led клубов.
- `server/routes/reading-sessions.ts` — reading session lifecycle, join/leave, listeners.
- `server/routes/questions.ts` — вопросы/комментарии эфира и отметка устно отвеченных вопросов.
- `server/routes/reactions.ts` — реакции слушателей с таймкодом.
- `server/routes/recordings.ts` и `server/services/recording-service.ts` — workflow записей и публикации.
- `server/websocket/reading-sessions.ts` — realtime listener/reaction/question события.

Клиент:

- `client/src/pages/reader-club-details.tsx` — отдельная страница клуба чтецов.
- `client/src/pages/club-route.tsx` — роутинг обычной/reader-led страницы клуба.
- `client/src/components/reader/ClubReader.tsx` — ридер и embedded Studio владельца.
- `client/src/components/studio/EmbeddedClubStudioShell.tsx` и `ControlBar.tsx` — панель Studio, реакции, вопросы, завершение эфира.
- `client/src/components/studio/ListenerOverlay.tsx` — слушательский overlay-плеер, реакции, вопросы.
- `client/src/hooks/use-club-live-listening.ts` — подключение слушателя к live-эфиру.
- `client/src/hooks/use-studio-device-eligibility.ts` — guard устройства/разрешения для Studio.
- `client/src/hooks/use-audio-session.ts` — Socket.IO события с REST fallback для live-индикаторов.

## Права доступа к книге

Ключевое правило: слушатель клуба чтецов не должен получать текст книги.

Для слушателей reader-led клуба закрыты:

- текст книги;
- файл книги;
- главы/контент ридера;
- клубная библиотека;
- заметки и закладки, связанные с текстом.

Ограничение реализуется серверно. UI-скрытие вкладки «Библиотека» и кнопок чтения — только UX-слой и не считается достаточной защитой.

## Заявки и приглашения

Поддерживаются два сценария попадания слушателя в клуб:

1. **Заявка на вступление** через карточку reader-led клуба.
   - Доступна авторизованному пользователю.
   - Создаёт pending membership.
   - Владелец получает личное сообщение.
   - При одобрении пользователь становится активным участником.
   - При отклонении пользователь не получает доступ и уведомляется через direct messages.

2. **Клубные приглашения** через существующую invitation-систему.
   - Зарегистрированный пользователь принимает invite и становится участником.
   - Незарегистрированный пользователь регистрируется по invite token; после регистрации создаётся членство в нужном клубе.
   - Приглашение не повышает роль выше слушателя.

## Live-аудио и слушательский плеер

Клубы чтецов используют текущий baseline VoxLibris Studio на Icecast/streaming route. Отдельный WebRTC-контур для первой версии не вводится.

Сценарий слушателя:

- слушатель видит состояние live-эфира на странице клуба;
- при подключении основной интерфейс клуба блюрится;
- поверх страницы открывается `ListenerOverlay`;
- Play/Pause слушателя управляет только локальным playback и не влияет на эфир для всех;
- доступны громкость, отключение от эфира, реакции и вопрос чтецу.

## Вопросы, комментарии и реакции эфира

Реакции слушателей:

- сохраняются в `session_reactions`;
- включают emoji, тип реакции и таймкод прослушивания;
- доставляются чтецу через `/reading-sessions` Socket.IO;
- дополнительно синхронизируются HTTP fallback в Studio;
- учитываются в `session_analytics`.

Вопросы/комментарии слушателей:

- отправляются через `POST /api/questions`;
- сохраняются в `session_questions`;
- увеличивают `questionCount` в аналитике;
- доставляются чтецу через Socket.IO и подтягиваются Studio через REST fallback `/api/questions/session/:sessionId/unanswered`;
- на панели Studio отображается только индикатор `?` со счётчиком неотвеченных вопросов;
- текст вопросов скрыт до клика на индикатор;
- чтец может ответить устно во время эфира и отметить вопрос галочкой;
- отметка галочкой вызывает `PUT /api/questions/:questionId/answer` и уменьшает счётчик неотвеченных.

## VoxLibris Studio и device guard

Studio доступна только владельцу reader-led клуба на подходящем устройстве:

- не мобильное устройство;
- не планшет;
- экран не меньше `1024 × 768`.

Сообщение пользователю при блокировке:

> Для нормальной работы в Voxlibris Studio необходимо разрешение экрана 1024 × 768. Пожалуйста, подключите соответствующий монитор

Guard является UX-слоем. Право запуска эфира также проверяется сервером.

## Записи эфиров

Текущий workflow записей:

- чтец может включить запись для будущей публикации;
- если публикационная запись не включена, техническая запись может сохраняться только для административного/арбитражного сценария;
- после завершения эфира запись проходит административную модерацию;
- чтец не получает запись до прохождения модерации;
- после модерации владелец клуба явно публикует запись;
- слушателям запись доступна только после модерации, публикации владельцем и будущей тарифной проверки.

Фактическая тарифная система для прослушивания/скачивания записей отложена.

## Пост-аналитика

Owner-only analytics блока клуба чтецов включает:

- количество эфиров с аналитикой;
- слушателей и среднее число слушателей;
- реакции;
- вопросы/комментарии;
- техническую оценку качества;
- пользовательские оценки и feedback;
- последние вопросы сессий и статус ответа.

Endpoint `/api/clubs/:clubId/reader-analytics` доступен владельцу reader-led клуба и системным admin/moderator. Слушатель получает отказ доступа.

## Лендинг и публичные страницы

- `/readers` — рейтинг/каталог чтецов.
- `/become-reader` — заявка на получение допуска ПРО-чтеца.
- На главном лендинге есть отдельные блоки reader-led клубов и топ-чтецов, управляемые feature flags:
  - `landing.readerClubs.enabled`;
  - `landing.topReaders.enabled`.
- Карточки reader-led клубов маркируются голосовым чтением и содержат компактную кнопку заявки для авторизованных пользователей.

Для Stage 12 добавлен smoke-скрипт:


Скрипт проверяет статические инварианты реализации и может выполнять HTTP-smoke против поднятого сервера через `STAGE12_BASE_URL` и тестовые токены.

Перед релизом рекомендуется пройти:

```bash
pnpm run test
pnpm run quality:gate
pnpm run stage12:reader-clubs
```

## Ограничения первой версии

Не входит в текущий production slice:

- сложная монетизация и тарифы;
- несколько чтецов/co-hosts в одном клубе;
- публичная витрина записей для слушателей;
- автоматический расчёт высокого рейтинга чтеца;
- автоматическая модерация аудиозаписей;
- отдельный WebRTC/mediasoup-контур для клубов чтецов.

Связанные миграции reader-led/recording workflow применяются вручную и строго последовательно, как все production-миграции VoxLibris:

```bash
ls migrations/ | sort | tail -5
```
