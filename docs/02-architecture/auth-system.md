# Система аутентификации и пользовательской системы

## Обзор

Модуль аутентификации и пользовательской системы - это центральная часть приложения VoxLibris, обеспечивающая регистрацию, аутентификацию, авторизацию и управление профилями пользователей. Система построена на основе JWT-токенов с поддержкой refresh-токенов для обеспечения безопасности и масштабируемости.

## Компоненты системы

### 1. Auth Routes ([server/auth-routes.ts](file:///home/odmen/DEV/voxlibris/server/auth-routes.ts))

Основной файл, содержащий маршруты аутентификации:

- `/api/auth/register` - регистрация нового пользователя
- `/api/auth/login` - вход пользователя в систему
- `/api/auth/logout` - выход из системы
- `/api/auth/refresh` - обновление access-токена с помощью refresh-токена
- `/api/auth/me` - получение информации о текущем пользователе
- `/api/auth/forgot-password` - запрос на восстановление пароля
- `/api/auth/reset-password` - сброс пароля по токену
- `/api/auth/activate/:token` - активация аккаунта по токену

### 2. Auth Service ([server/auth-service.ts](file:///home/odmen/DEV/voxlibris/server/auth-service.ts))

Сервис, реализующий бизнес-логику аутентификации:

- Генерация и верификация JWT-токенов
- Хеширование паролей с использованием bcrypt
- Хранение и проверка refresh-токенов в базе данных
- Управление сессиями пользователей
- Периодическая очистка просроченных токенов

### 3. JWT Middleware ([server/jwt-middleware.ts](file:///home/odmen/DEV/voxlibris/server/jwt-middleware.ts))

Промежуточное ПО для проверки JWT-токенов:

- Извлечение токенов из заголовков или cookies
- Верификация токенов
- Установка информации о пользователе в req.user
- Проверка статуса активности пользователя

## Архитектура аутентификации

### JWT-токены

Система использует два типа токенов:

1. **Access Token**:
   - Время жизни: короткое (обычно 15-30 минут)
   - Используется для доступа к защищенным маршрутам
   - Хранит минимальную информацию о пользователе

2. **Refresh Token**:
   - Время жизни: длинное (обычно 7-30 дней)
   - Используется для получения новых access-токенов
   - Хранится в базе данных с возможностью отзыва
   - Может быть использован только один раз

### Процесс аутентификации

```
1. Пользователь отправляет учетные данные (email/username и пароль)
2. Сервер проверяет учетные данные в базе данных
3. При успешной проверке:
   a. Генерируется новый access-токен
   b. Генерируется новый refresh-токен
   c. Refresh-токен сохраняется в базе данных
   d. Токены отправляются пользователю (access-токен в заголовке, refresh-токен в httpOnly cookie)
4. Пользователь использует access-токен для доступа к защищенным ресурсам
5. При истечении access-токена, пользователь использует refresh-токен для получения новых токенов
```

### Безопасность

- Пароли хешируются с использованием bcrypt с 12 раундами
- Refresh-токены хранятся в базе данных и могут быть отозваны
- Access-токены содержат минимальную информацию и имеют короткий срок действия
- HttpOnly cookies защищают от XSS-атак
- Реализована защита от brute-force атак через rate limiting

## Основные функции

### Регистрация пользователя

1. Проверка уникальности email и username
2. Валидация данных с использованием Zod
3. Хеширование пароля
4. Создание записи пользователя в базе данных
5. Генерация токенов
6. Отправка email для подтверждения аккаунта

### Вход в систему

1. Проверка учетных данных
2. Проверка статуса активации аккаунта
3. Генерация новой пары токенов
4. Сохранение refresh-токена в базе данных
5. Отправка токенов пользователю

### Обновление токенов

1. Проверка refresh-токена в базе данных
2. Проверка валидности токена
3. Генерация новой пары токенов
4. Удаление старого refresh-токена из базы
5. Сохранение нового refresh-токена
6. Отправка нового access-токена пользователю

### Восстановление пароля

1. Пользователь указывает email для восстановления
2. Генерируется временный токен восстановления
3. Токен сохраняется в базе данных
4. Отправляется email с ссылкой для сброса пароля
5. При переходе по ссылке и вводе нового пароля:
   - Токен проверяется на валидность
   - Пароль хешируется и обновляется
   - Все refresh-токены пользователя аннулируются

## Интеграция с другими модулями

### Middleware

JWT middleware используется во многих маршрутах для проверки аутентификации:

```typescript
app.use("/api/clubs", jwtAuth, clubRoutes);
app.use("/api/v1/books", jwtAuth, readerRoutes);
```

### Пользовательские права

Система аутентификации также предоставляет информацию о ролях пользователей:

- `user` - обычный пользователь
- `moderator` - модератор (ограниченные административные функции)
- `admin` - администратор (полный доступ к системе)

### Профили пользователей

Информация о пользователях используется в других модулях:

- В клубах для проверки участия
- В книгах для отслеживания владельцев
- В сессиях чтения для определения участников

## Возможные проблемы и решения

### Проблемы с токенами

- **Refresh-токен истек**: пользователь должен снова выполнить вход
- **Access-токен истек**: автоматическое обновление с помощью refresh-токена
- **Refresh-токен скомпрометирован**: все токены пользователя аннулируются

### Rate Limiting

Система имеет встроенную защиту от подбора паролей:

- Ограничения на количество попыток входа
- Увеличение задержки после нескольких неудачных попыток
- Блокировка IP-адреса при подозрительной активности

## Тестирование

Модуль аутентификации должен быть протестирован на:

- Корректность регистрации и входа
- Безопасность токенов
- Обработку ошибок
- Защиту от атак
- Интеграцию с другими частями приложения
