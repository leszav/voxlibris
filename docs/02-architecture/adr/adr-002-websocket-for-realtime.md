# ADR-002: Использование WebSocket для синхронизации в реальном времени

## Название

Использование WebSocket для синхронизации сессий чтения в реальном времени

## Статус

Принято

## Контекст

В приложении VoxLibris требуется синхронизация прогресса чтения между участниками сессии в реальном времени, а также поддержка чата и других интерактивных функций во время сессии. Необходимо выбрать технологию для двунаправленной связи в реальном времени между клиентом и сервером.

Были рассмотрены следующие альтернативы:
- WebSocket (с использованием Socket.IO)
- Server-Sent Events (SSE)
- Long polling
- WebRTC (для peer-to-peer соединений)
- GraphQL subscriptions

## Решение

Мы выбираем WebSocket с использованием библиотеки Socket.IO для синхронизации в реальном времени по следующим причинам:

1. **Двунаправленная связь**: WebSocket обеспечивает двустороннюю связь в реальном времени, что критично для синхронизации прогресса чтения и обмена сообщениями в чате.

2. **Низкая задержка**: WebSocket имеет минимальную задержку после установления соединения, что важно для синхронизации действий участников сессии.

3. **Поддержка комнат**: Socket.IO предоставляет встроенную поддержку комнат, что идеально подходит для сессий чтения с ограниченным числом участников.

4. **Надежность**: Socket.IO включает механизмы восстановления соединения и fallback для старых браузеров.

5. **Интеграция**: Хорошо интегрируется с существующим стеком (Express.js, React).

6. **Масштабируемость**: Поддерживает масштабирование с использованием Redis Adapter для кластеров.

## Последствия

### Положительные эффекты

- Возможность мгновенной синхронизации прогресса чтения
- Поддержка интерактивного чата во время сессии
- Возможность отправки реакций в реальном времени
- Низкая задержка для управления воспроизведением

### Негативные эффекты

- Необходимость управления долгосрочными соединениями на сервере
- Увеличение сложности при масштабировании (необходим Redis для кластеров)
- Потребность в обработке отключения клиентов и восстановления соединений
- Увеличение потребления ресурсов по сравнению с HTTP

### Отказ от других альтернатив

- **Server-Sent Events**: Отклонен из-за односторонней связи (только сервер → клиент)
- **Long polling**: Отклонен из-за высокой нагрузки на сервер и задержек
- **WebRTC**: Отклонен из-за сложности реализации и несоответствия сценариям использования
- **GraphQL subscriptions**: Отклонен из-за дополнительной сложности и отсутствия необходимости в GraphQL
