# ADR-006: Архитектура системы клубов и сообществ

## Название

Использование многоуровневой архитектуры с иерархией ролей для системы клубов и сообществ

## Статус

Принято

## Контекст

В платформе VoxLibris необходимо реализовать систему клубов и сообществ, позволяющую пользователям создавать, управлять и участвовать в книжных клубах. Система должна поддерживать:

- Различные типы клубов (открытые, закрытые, с ограниченным доступом)
- Иерархию ролей участников (владелец, модератор, участник, гость)
- Механизмы приглашений
- Управление контентом клуба
- Механизмы модерации

Были рассмотрены следующие альтернативы:
- Простая модель без ролей
- Модель с гибкими ролями
- Модель с фиксированной иерархией ролей

## Решение

Мы выбираем архитектуру с фиксированной иерархией ролей и поддержкой различных типов клубов по следующим причинам:

1. **Четкое определение прав**: Фиксированная иерархия ролей (владелец → модератор → участник → гость) позволяет точно определить права доступа.

2. **Масштабируемость**: Такая модель позволяет легко масштабировать платформу до крупных сообществ.

3. **Безопасность**: Четкие границы ролей предотвращают несанкционированный доступ к функциям управления.

4. **Простота реализации**: Модель легче реализовать и тестировать, чем систему с гибкими ролями.

5. **UX**: Пользователи легче понимают свои права и возможности в клубе.

## Последствия

### Положительные эффекты

- Легко понимаемая модель для пользователей
- Упрощенное управление правами доступа
- Легко тестируемая система
- Поддержка различных сценариев использования (от личных клубов до крупных сообществ)

### Негативные эффекты

- Ограниченная гибкость по сравнению с системой с полностью настраиваемыми ролями
- Потребность в механизмах обхода ограничений для специфических случаев

### Отказ от других альтернатив

- **Простая модель без ролей**: Отклонена из-за невозможности обеспечить безопасность и управление
- **Модель с гибкими ролями**: Отклонена из-за сложности реализации и потенциальных проблем с безопасностью
