# ADR-003: Использование JWT для аутентификации

## Название

Использование JWT (JSON Web Tokens) для аутентификации пользователей

## Статус

Принято

## Контекст

В приложении VoxLibris требуется безопасный и масштабируемый механизм аутентификации пользователей. Нам нужно выбрать подход для аутентификации, который:

- Обеспечивает безопасность передачи данных
- Позволяет масштабировать приложение (stateless архитектура)
- Поддерживает как веб-браузеры, так и мобильные приложения
- Совместим с существующим стеком (React, Express)
- Обеспечивает приемлемый UX (возможность "запомнить меня", автоматический refresh токенов)

Были рассмотрены следующие альтернативы:
- Традиционные сессии на сервере (с хранением в Redis/DB)
- JWT токены с refresh токенами
- OAuth 2.0 (через внешние провайдеры)
- Custom токены

## Решение

Мы выбираем JWT токены в сочетании с refresh токенами, хранящимися в httpOnly cookie, для аутентификации пользователей по следующим причинам:

1. **Stateless архитектура**: JWT позволяют не хранить состояние сессии на сервере, что упрощает масштабирование.

2. **Безопасность**: Использование httpOnly cookies для refresh токенов защищает от XSS атак, а короткоживущие access токены снижают риск при компрометации.

3. **Совместимость**: JWT стандарт де-факто для аутентификации в современных веб-приложениях, поддерживается большинством фреймворков.

4. **Гибкость**: JWT может содержать произвольные данные (роли, права доступа), что упрощает авторизацию на уровне приложения.

5. **UX**: Refresh токены позволяют поддерживать длительные сессии без частого повторного входа, при этом обеспечивая безопасность.

## Последствия

### Положительные эффекты

- Упрощенное масштабирование (без необходимости хранения сессий)
- Упрощенная архитектура (без необходимости централизованного хранения сессий)
- Совместимость с microservice архитектурой
- Лучший UX при работе с приложением

### Негативные эффекты

- Невозможность преждевременно инвалидировать токен (пока не истечет)
- Увеличенный размер заголовков HTTP (по сравнению с сессионными ID)
- Необходимость защиты от replay атак
- Сложнее обработка ситуаций с несколькими активными сессиями

### Отказ от других альтернатив

- **Традиционные сессии**: Отклонены из-за усложнения масштабирования и необходимости централизованного хранения
- **OAuth 2.0**: Отклонен как слишком сложный для основного метода аутентификации, может использоваться как дополнительный метод
- **Custom токены**: Отклонены из-за отсутствия стандартов и необходимости дополнительной разработки
