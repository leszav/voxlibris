# ADR-005: Выбор PostgreSQL в качестве основной базы данных

## Название

Выбор PostgreSQL в качестве основной реляционной базы данных для хранения данных

## Статус

Принято

## Контекст

Для хранения структурированных данных VoxLibris необходимо выбрать реляционную базу данных. Требования к базе данных:

- Поддержка сложных запросов и связей между сущностями
- Высокая надежность и стабильность
- Поддержка JSON и других типов данных для гибкости
- Масштабируемость (горизонтальная и вертикальная)
- Поддержка транзакций и ACID свойств
- Совместимость с Drizzle ORM
- Хорошая производительность
- Наличие инструментов для мониторинга и администрирования

Были рассмотрены следующие альтернативы:
- PostgreSQL
- MySQL
- MariaDB
- SQLite (для малых установок)
- MongoDB (для документоориентированного подхода)

## Решение

Мы выбираем PostgreSQL в качестве основной реляционной базы данных по следующим причинам:

1. **Расширенные возможности**: PostgreSQL предоставляет более широкие возможности для сложных запросов, агрегаций и аналитики по сравнению с MySQL.

2. **Поддержка JSON**: Встроенная поддержка JSON и JSONB типов данных позволяет хранить полуструктурированные данные, что полезно для хранения метаданных книг и настроек пользователей.

3. **Расширяемость**: PostgreSQL легко расширяется пользовательскими типами данных, операторами и функциями.

4. **Производительность**: PostgreSQL показывает хорошую производительность при сложных запросах и большом объеме данных.

5. **Безопасность**: Сильные возможности по управлению доступом и безопасности.

6. **Совместимость с Drizzle ORM**: Хорошая поддержка через @libsql/client и другие драйверы.

7. **Открытость и сообщество**: PostgreSQL полностью open-source с активным сообществом разработчиков и пользователей.

## Последствия

### Положительные эффекты

- Поддержка сложных запросов и аналитики
- Гибкость хранения полуструктурированных данных
- Высокая надежность и стабильность
- Хорошая производительность при правильно настроенных индексах
- Возможность горизонтального масштабирования через внешние инструменты

### Негативные эффекты

- Потенциально более высокое потребление памяти по сравнению с MySQL
- Более крутая кривая обучения для администраторов
- Меньше хостинг-провайдеров с встроенной поддержкой

### Отказ от других альтернатив

- **MySQL**: Отклонен из-за менее развитой поддержки JSON и расширяемости
- **MariaDB**: Отклонен как форк MySQL с ограниченной экосистемой
- **SQLite**: Отклонен из-за ограничений масштабируемости и многопользовательского доступа
- **MongoDB**: Отклонен из-за отсутствия строгой схемы и несоответствия реляционной модели данных приложения
